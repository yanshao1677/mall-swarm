# 下单事务内锁库存与支付后扣减

## 本项目落地状态

| 维度 | 状态 |
|------|------|
| 代码位置 | `mall-portal/OmsPortalOrderServiceImpl.generateOrder/paySuccess/cancelOrder` |
| 默认是否启用 | ✅ 主流程已启用 |
| 已实现支付 | 支付宝（`AlipayController`）；`payType=2` 微信 **仅注释，无实现** |
| 已知缺陷 | 直接调 `paySuccess(orderId)` **不校验** status，可能重复扣库存；应用 `paySuccessByOrderSn` |

## 1. 问题

下单与支付异步：需在下单时预留库存防超卖，未支付可释放，支付成功才扣真实库存。

## 2. 适用约束

- SKU 表有 `stock`（可售）与 `lock_stock`（锁定）字段
- 下单 `@Transactional`
- 支付回调或主动确认后才 `paySuccess`

## 3. 核心思路

**两阶段**：`generateOrder` 内 `lock_stock += quantity`（stock 不变）；`paySuccess` 内 `stock -= quantity` 且 `lock_stock -= quantity`；取消/超时仅 `lock_stock -= quantity`（releaseSkuStockLock）。

## 4. 处理流程

1. 校验 `realStock = stock - lock_stock >= quantity`
2. 事务内 lockStock → 插订单 status=0 → 发延迟取消 MQ
3. paySuccess：status=1，updateSkuStock 扣减
4. cancelOrder：status=4，releaseSkuStockLock

## 5. 异常处理

- 库存不足：下单前 `Asserts.fail("库存不足，无法下单")`，事务回滚
- 支付直接 API `paySuccess`：**不校验** status，重复调用可能重复扣 stock（风险点）
- 取消幂等：status≠0 不释放

## 6. 具体语言实现

```java
// 下单锁库存（DAO 层典型：UPDATE sku SET lock_stock = lock_stock + #{qty} WHERE id=#{id} AND stock-lock_stock >= #{qty}）
lockStock(cartPromotionItemList);

// 支付扣减
portalOrderDao.updateSkuStock(orderItemList); // stock--, lock_stock--

// 取消释放
portalOrderDao.releaseSkuStockLock(orderItemList); // lock_stock--
```

## 7. 测试点

- 下单后 lock_stock 增、stock 不变
- 支付后两者同减
- 超时取消后 lock_stock 回退
- 并发下单超卖：依赖 SQL 条件或乐观锁
