# 支付回调按订单号幂等确认

## 本项目落地状态

| 维度 | 状态 |
|------|------|
| 代码位置 | `paySuccessByOrderSn` + `AlipayServiceImpl.notify` |
| 逻辑是否实现 | ✅ `status=0` 闸门 + 静默跳过已支付 |
| 默认可跑通 | ⚠️ `config/portal/mall-portal-dev.yaml` 中 `alipay.notifyUrl` **常为空**，异步回调需自行配置公网 URL |
| 未实现 | 微信支付回调 |

## 触发条件

- 支付宝异步 `POST /alipay/notify` 验签通过且 `trade_status == TRADE_SUCCESS`
- 或主动查询 `GET /alipay/query` 返回 TRADE_SUCCESS
- 调用 `paySuccessByOrderSn(outTradeNo, payType)`，payType=1（支付宝）

## 输入字段

| 字段 | 类型 | 来源 |
|------|------|------|
| orderSn | String | 支付宝 `out_trade_no`（下单时 generateOrderSn 生成） |
| payType | Integer | 1=支付宝 |

## 判定规则

**查询条件（必须同时满足才执行 paySuccess）**：

```
orderSn = 参数 orderSn
status = 0
deleteStatus = 0
```

```
IF orderList 非空:
    paySuccess(order.getId(), payType)
ELSE:
    无操作（静默）
```

`paySuccess` 内：
- `status = 1`，`paymentTime = now`
- `updateSkuStock`：`stock -= qty`，`lock_stock -= qty`

**对比**：直接 `paySuccess(orderId, payType)` **无** status 前置校验，重复调用可能重复扣库存。

## 状态读写位置

- 读/写：oms_order
- 写：pms_sku_stock

## 正常路径

1. notify 验签
2. trade_status 判断
3. paySuccessByOrderSn
4. 查 status=0 订单
5. paySuccess 扣库存
6. 返回支付宝 `"success"`

## 分支路径

- 验签失败 → `"failure"`，支付宝重试
- trade_status 非 SUCCESS → warn 日志 + `"failure"`
- 重复 notify 同一 orderSn → 第二次查不到 status=0 → 静默
- 订单已超时关闭 status=4 → 查不到 → 静默（**证据不足**：是否与业务期望「已关闭仍收款」冲突需产品确认）

## 失败处理

- paySuccess 内 DB 失败 → 异常；支付宝重试 → 幂等闸门保护

## 幂等性 / 一致性约束

- 以 **status=0** 为支付确认闸门
- 不处理并发双支付竞态（两回调同时到需 DB 乐观锁，证据不足：update 是否带 status 条件）

## 代码骨架

```java
public void paySuccessByOrderSn(String orderSn, Integer payType) {
    OmsOrderExample ex = new OmsOrderExample();
    ex.createCriteria()
        .andOrderSnEqualTo(orderSn)
        .andStatusEqualTo(0)
        .andDeleteStatusEqualTo(0);
    List<OmsOrder> list = orderMapper.selectByExample(ex);
    if (CollUtil.isNotEmpty(list)) {
        paySuccess(list.get(0).getId(), payType);
    }
}
```

## 最小验证清单

- 待付款订单首次 notify → status=1，库存扣减
- 同一 orderSn 第二次 notify → 订单保持 status=1，库存不重复扣
- 不存在 orderSn → 静默无异常
