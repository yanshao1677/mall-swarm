# 开源项目工程经验提炼提示词模板

> **目标**：把读过的开源项目沉淀成一个中立的工程经验库。大到整体架构与设计取舍，小到某个具体方案的处理方式，都能查。
>
> **铁律**：经验文档正文**完全脱离原项目**，一个没见过该仓库的人或 AI，只读这篇文档就能理解问题、看懂方案、自己写出来。来源证据（原项目路径/函数名）只放附录用于溯源核实，不进正文。
>
> **核心流程**：先总纲，后深读；先事实，后经验；先审查，后入库；用时先检索，再让 AI 编程工具落地。

经验分**三层**。层级越往下越具体、越能抵抗“大概/应该”的空洞表达：

| 层级 | 文档类型 | 回答什么 | 重点 | 颗粒度 |
|------|----------|----------|------|--------|
| 架构层 | 架构决策记录（ADR） | 整体为什么这样搭、为什么选 A 不选 B、核心抽象是什么 | 背景、权衡、适用边界 | 一个架构决策 |
| 方案层 | 方案卡（Pattern） | 重试怎么做、状态怎么持久化、检索怎么融合 | 机制、可复用结构 | 一个完整方案（含多个阶段） |
| 原子层 | 原子规格卡（Atomic Spec） | 这一步精确怎么处理、参数为何是这个数、出错怎么办 | 可直接执行的精确规格 | 一个决断点 / 一个处理阶段 |

> **空洞的根因是颗粒度太大。** 范围一大，AI 就退回概括，概括即空洞。对抗它的唯一办法是把提炼单元砍到原子级——精确到字段、参数、阈值、分支、失败动作。一张原子卡只回答一个“为什么是这样而不是那样”，逼到 AI 要么交出代码里的确切答案，要么诚实写“证据不足”。
>
> 方案卡（场景 4B）是承载多张原子卡的容器；真正干货在**场景 4C 的原子规格卡**。

适用于 Codex / Claude Code / Cursor / OpenCode 等 AI 编程工具。

## 0. 固定前置提示词：防偷懒规则

每次任务前先贴这一段。

```txt
你现在不是普通聊天助手，而是工程项目阅读和经验提炼助手。

防偷懒规则：

1. 不允许只根据 README、文件名或常识做判断。
2. 每个关键结论必须附带证据：文件路径、类名、函数名、配置项、文档位置。
3. 所有引用的文件路径、类名、函数名必须是你实际打开或检索过的：
   - 禁止根据命名惯例推测路径。
   - 禁止编造看似合理但未验证的函数名或类名。
   - 未实际验证的必须标“路径待核实”。
   - 引用关键结论时附带行号或 grep 关键词，方便人工复查。
4. 没有证据的内容必须标“推测”，不能写成确定结论。
5. 必须区分：已确认 / 推测 / 未阅读 / 待深读。
6. 禁止空泛表达（架构清晰、高内聚低耦合、易扩展、工程化良好），除非说明具体代码机制。
7. 禁止用模糊词糊弄核心结论。以下词出现在关键判定里即视为没读懂、必须打回重读：
   大概、应该、可能、通常会、一般来说、整体上、差不多、类似地处理、做了相应处理。
   核心结论必须落到：确切的值、确切的字段名、确切的分支条件、确切的失败动作。
   确实读不到的，写“证据不足：需确认 X 是 A 还是 B”，绝不脑补。
8. 阅读范围过大时必须主动拆分任务，不要强行总结。
9. 输出末尾必须包含：已阅读文件列表、未阅读但可能重要的文件列表、下一步建议深读文件。
10. 你的目标不是生成项目介绍，而是提炼可脱离原项目、可复用、能指导写代码的工程经验。
```

## 1. 场景一：第一次读开源项目，先生成总纲

刚 clone 一个项目后使用。此时只建地图，不提炼经验。

```txt
你现在是“开源项目阅读总控”。

目标：阅读当前项目，建立项目总纲。现在不要提炼经验，不要急着给结论。

阅读优先级：
1. README   2. docs   3. examples
4. 配置文件（package.json / pyproject.toml / pom.xml / go.mod 等）
5. 入口文件   6. 核心目录结构（src / app / packages 等）
7. tests 中能体现核心流程的测试

请输出：

## 1. 项目背景
解决什么问题？面向什么用户？典型使用场景？

## 2. 技术栈
主要语言、框架、中间件、工具。

## 3. 核心模块地图
不超过 10 个核心模块，每个说明：职责 / 关键文件路径 / 与其他模块关系 / 是否值得深读。

## 4. 整体架构判断（为后续 ADR 做准备）
- 这个项目最核心的抽象是什么？
- 它的分层 / 模块边界是怎么划的？
- 有哪些一眼可见的重大设计取舍（同步异步、单体微服务、插件化、事件驱动等）？
- 这些先记为“待深读的架构问题”，不要急着下结论。

## 5. 主流程识别
找出 3~5 条最值得深读的主流程（如请求处理、任务执行、工具注册调用、记忆读写、workflow 流转、失败重试）。

## 6. 阅读优先级
P0 必须读 / P1 建议读 / P2 暂时可不读。

## 7. 暂不阅读范围
哪些目录暂时不读？为什么？

## 8. 已确认事实
只列有证据的。

## 9. 推测和未确认问题

## 10. 下一步任务建议
拆成 3~8 个子任务，每个只聚焦一个模块、一条流程或一个架构问题。

要求：每个判断带证据，未验证的标“路径待核实”，没证据的标“未确认”。
```

## 2. 场景二：深读单个模块

只读 memory、tool registry、workflow、RAG、auth、plugin 等单个模块。

```txt
你现在是“模块深读 Agent”。本次只读一个模块，不扩散到整个项目。

目标模块：【填写，例如 tool registry / memory / workflow / RAG / auth / plugin】

阅读范围：该模块源码、相关测试、相关 examples、相关 docs、该模块被调用的入口。

请输出：

## 1. 模块背景      这个模块解决什么问题？
## 2. 核心职责      负责什么？不负责什么？
## 3. 核心对象      主要类/函数/接口/数据结构，每个说明：作用 / 文件路径（实际打开过的，否则标“路径待核实”）/ 与其他对象关系。
## 4. 内部流程      按步骤说明运行流程：输入来源、校验、分发、执行、返回、异常处理。
## 5. 对外接口      暴露了什么能力？
## 6. 扩展点        新增类似能力要改哪里？新增功能的路径是什么？
## 7. 错误处理      异常、失败、重试、兜底怎么做的？
## 8. 设计优点      必须基于代码证据，不允许空泛评价。
## 9. 设计代价      带来了哪些复杂度？
## 10. 可提炼候选   列 3~10 条候选，并标注每条属于“架构层”还是“方案层”。先只列候选，不入库。
## 11. 已读证据     文件、类、函数，附行号或 grep 关键词。
## 12. 待深读问题

禁止：总结整个项目 / 无证据结论 / 空话 / 编造未验证的路径或函数名。
```

## 3. 场景三：深读一条业务/技术流程

读“Agent 执行一次任务”“OCR 到入库”“检索到回答”“工具调用完整链路”等。

```txt
你现在是“流程深读 Agent”。只分析下面这条流程，不扩散到其他模块。

目标流程：【填写，例如 一次任务从用户输入到工具调用再到结果返回】

从代码追踪完整链路：

## 1. 流程入口      从哪个文件/函数/接口开始？
## 2. 输入数据      输入是什么？数据结构？关键字段？
## 3. 中间步骤      按真实执行顺序拆解，每步说明：发生了什么 / 调用了哪个函数类 / 输入输出 / 文件路径（实际验证过的）。
## 4. 状态变化      有哪些状态？如何变化？存在哪里？是否支持暂停、恢复、重试？
## 5. 分支逻辑      有哪些 if/else、策略选择、路由分发、异常分支？
## 6. 外部依赖      调用了哪些外部服务、数据库、缓存、模型、工具？
## 7. 失败处理      重试、回滚、降级、日志？
## 8. 输出结果      最终输出是什么？如何返回上层？
## 9. 流程图        用 Mermaid 画简洁流程图。
## 10. 可提炼候选   这条流程里哪些设计值得提炼？标注每条属“架构层 / 方案层 / 原子层”。
                    对值得深挖的点，建议直接拆成多个原子阶段（阶段1/阶段2/…），每个阶段一张原子规格卡。
## 11. 证据列表     关键文件、类、函数，附行号或 grep 关键词。

要求：按代码真实链路追踪，不确定标“未确认”，未验证路径标“路径待核实”。
```

## 4A. 场景四 A：提炼【架构层】经验 —— 架构决策记录（ADR）

适合把“整体怎么搭的、为什么这么选”沉淀下来。**正文必须脱离原项目。**

```txt
你现在是“架构经验提炼 Agent”。

目标：从总纲和深读笔记中，提炼一条可脱离原项目阅读的架构决策记录。
你不是介绍这个项目，而是把它的某个架构选择抽象成一条任何人都能参考的决策经验。

脱离原项目硬性自检（不过则不准入库）：
- 删掉所有原项目专有名词后，这篇文档是否仍然完整可读？
- 一个没见过原仓库的人，只读这篇，能否理解这个架构决策的来龙去脉？
- 能否据此在一个全新项目里做出类似的架构选择？

每条 ADR 用下面格式：

# 架构决策标题（用通用语言，不带原项目名）

## 1. 背景与场景
这个决策是在什么样的业务/工程场景下需要做的？（描述场景类型，不描述具体项目）

## 2. 要解决的核心问题
不做这个决策、或做错，会出现什么后果？

## 3. 可选方案
列出当时合理的几个候选方案（至少 2 个），各自的代表性做法。

## 4. 决策与理由
最终选了哪个？为什么？关键权衡点是什么（性能/复杂度/可维护性/团队成本/扩展性）？

## 5. 核心抽象
这个架构里最关键的抽象是什么？用一段不含原项目名的话说清。

## 6. 通用结构图
用 Mermaid 画出脱离原项目的通用架构示意（模块/层/边界）。

## 7. 适用条件
什么规模、什么团队、什么约束下，这个决策成立？

## 8. 不适用 / 反例
什么情况下这个决策是过度设计或错误选择？

## 9. 已知代价
选了它要长期承担哪些复杂度或限制？

## 10. 落地要点
要复现这个架构，最关键的几步是什么（通用，不绑技术栈）？

## 11. 标签
如：architecture, layering, event-driven, plugin, async, multi-tenant, state-machine

## 12. 来源证据（附录，不进正文，仅供溯源）
原项目仓库 / 文件路径 / 类名 / commit。未验证的标“路径待核实”。

质量要求：
1. 正文出现任何原项目专有名词都算不合格。
2. 必须说清“为什么选它、放弃了什么”，只说“它很好”不合格。
3. 证据不足标“待深读”，不要硬编。
```

## 4B. 场景四 B：提炼【方案层】经验 —— 方案卡（Pattern）

适合把“某个具体问题怎么解的”沉淀下来。**正文脱离原项目，代码给具体语言实现。**

```txt
你现在是“方案经验提炼 Agent”。

目标：从模块/流程深读笔记中，提炼一条可脱离原项目阅读的方案卡。
你不是复述原项目代码，而是把它解决某个具体问题的方式，抽象成一个通用方案。

脱离原项目硬性自检（不过则不准入库）：
- 删掉原项目专有名词后，方案是否仍然自洽完整？
- 没见过原仓库的 AI，只读这张卡，能否写出可运行的实现？

每张方案卡用下面格式：

# 方案标题（用通用语言描述要解决的问题，不带原项目名）

## 1. 问题
要解决的具体问题是什么？不解决会怎样？

## 2. 适用约束
成立需要哪些前提（数据量、并发、是否需审计、是否人工确认、是否 AI 自动调用等）？

## 3. 核心思路
用一句话说清这个方案的机制本质（必须含机制，禁止只用形容词）。

## 4. 通用结构
脱离原项目，抽象出通用模块/对象/数据结构（不绑语言）。

## 5. 处理流程
按步骤说明方案如何运行，含正常路径与关键分支。

## 6. 异常处理
失败、重试、回滚、降级、幂等怎么做？

## 7. 具体语言实现
给出至少一种你有把握的语言的可运行实现（不是伪代码，是能跑的代码骨架）。
如果对某语言没把握，写“此方案在 X 语言中的实现需另行验证”，不要硬填。

## 8. 测试点
必须验证哪些关键行为（含边界与失败场景）？

## 9. 适用场景 / 不适用场景
什么时候用，什么时候是过度设计。

## 10. 风险与反模式
容易踩的坑。

## 11. 标签
如：retry, idempotency, state-persistence, rag-fusion, tool-registry, sse-streaming

## 12. 来源证据（附录，不进正文，仅供溯源）
原项目仓库 / 文件路径 / 类名 / 函数名 / commit。未验证的标“路径待核实”。

质量要求：
1. 正文出现原项目专有名词算不合格。
2. “核心思路”说不清机制算不合格。
3. 代码必须是能直接参考落地的实现，不是占位伪代码。
```

## 4C. 场景四 C：提炼【原子层】经验 —— 原子规格卡（Atomic Spec）★核心

这是对抗“空洞总结”的主力。一张原子卡只锁定**一个决断点或一个处理阶段**，逼 AI 交出精确到参数、字段、分支、失败动作的规格——而不是概括。一个方案（场景 4B）通常拆成多张原子卡，按“阶段 1 / 阶段 2 / 阶段 3”串起来。

> 判断颗粒度是否够小：如果这张卡里还能再切出一个独立的“为什么是这样而不是那样”，就说明还不够原子，继续切。

```txt
你现在是“原子规格提炼 Agent”。

目标：把一个具体的处理阶段或决断点，提炼成一张精确到可直接照着写代码的原子规格卡。
你不是描述这一步“做了什么”，而是规定它“精确怎么做、参数为何是这个值、出错走哪条分支”。

绝对禁止的表达（出现即作废、必须重读源码）：
大概、应该、可能、通常、一般来说、整体上、做了相应处理、类似地清洗、对数据进行校验……
每一条规则都必须落到：确切字段名、确切计算式、确切阈值、确切分支条件、确切失败动作。
读不到的，写“证据不足：需确认是 A 还是 B”，绝不脑补。

本次只提炼一个阶段/决断点：
【填写，例如：事件入口的短期去重判定】

请严格按下面结构输出（缺项就显式写“无”或“证据不足”）：

# 阶段标题（通用语言，不带原项目名）

## 触发条件
什么前置条件满足时，才进入这一步？（精确到具体状态/配置开关）

## 输入字段
逐个列出这一步读取的字段：字段名、类型、是否可选、来源。

## 判定规则
这是核心。逐条写出精确规则，必须包含：
- 关键变量怎么算（给出确切表达式，如 key = sha256(a + ":" + b)）
- 截断/规整规则（如 input.slice(0, 500)）
- 阈值与窗口的确切数值（如 TTL = 5 分钟，清理周期 = 60 秒）
- 默认值与回退（如 toolName = payload.tool_name ?? eventType）
对每个“魔法数字/阈值”追问一句：为什么是这个值？读不到就标“证据不足：阈值来源未确认”。

## 状态读写位置
这一步读/写什么状态？存在哪里（进程内 Map / Redis / DB / 无）？
若实现要跨进程，必须说明语义如何变化（如：进程内 Map 换成 Redis SET NX PX，否则去重语义不成立）。

## 正常路径
按真实顺序编号列出每一步动作。注意写清“先后顺序”本身的约束（如：核心写入成功后才记录去重 key）。

## 分支路径
列出所有分支及各自的确切行为：
- 命中且未过期 → 返回什么、跳过哪些后续阶段
- 命中但已过期 → 删除旧值、继续
- 各种输入异常 → 走哪条
每个分支都要有明确去向，不许写“做相应处理”。

## 失败处理
这一步可能怎么失败？失败时是回退、跳过、还是中断？
读不到确切答案的，写“证据不足：失败时是回退为 String(x) 还是跳过，需实现时确认”。

## 幂等性 / 一致性约束
这一步提供的是哪种级别的保证（近似/严格/审计级）？有什么不能依赖的前提（如：进程重启后缓存丢失，不能当审计级幂等）？
对后续阶段有什么强制约束（如：后续只能用清洗后的对象，禁止再读原始 payload）？

## 代码骨架
给出至少一种有把握语言的可运行实现（非伪代码）。没把握的语言写“需另行验证”。

## 最小验证清单
列出能证明这一步正确的最小测试用例，每条是可执行的断言，例如：
- 同一 key 在窗口内第二次调用返回 deduplicated=true
- 核心写入失败时不记录 key，重试仍会进入写入
- TTL 过期后同一 key 可再次写入

## 来源证据（附录，不进正文）
原项目文件路径 / 类名 / 函数名 / 行号。未验证标“路径待核实”。
```

**一张合格原子卡的样例骨架**（说明颗粒度，不是内容范本）：

```txt
# 阶段 2：事件入口的短期重复判定
触发条件：阶段 1 通过，且入口配置了短期去重缓存。
输入字段：streamId(string) / eventType(string) / payload.tool_name(string,可选) / payload.tool_input(any,可选)
判定规则：
  toolName  = payload.tool_name ?? eventType
  inputText = typeof tool_input === "string" ? tool_input.slice(0,500) : JSON.stringify(tool_input ?? "").slice(0,500)
  dedupKey  = sha256(streamId + ":" + toolName + ":" + inputText)
  TTL = 5 分钟；清理周期 = 60 秒
状态读写位置：进程内 Map<dedupKey, expiresAt>；跨进程需换 Redis SET NX PX，否则语义不成立。
正常路径：算 key → 读缓存 → 未命中/已过期则继续阶段 3 → 核心写入成功后才记 key。
分支路径：命中未过期 → 返回 {deduplicated:true}，跳过清洗/持久化/广播；命中已过期 → 删旧 key 继续。
失败处理：证据不足——JSON.stringify 遇循环结构失败时，是回退 String(input) 还是跳过去重，需实现确认。
幂等性：近似短期幂等；进程重启缓存丢失，不可作审计级幂等。
最小验证清单：①窗口内同 key 第二次返回 deduplicated ②写入失败不记 key、重试仍写入 ③TTL 过期后可再写。
```

> 注意样例里那句“失败处理：证据不足……需实现确认”——**这正是诚实的原子卡该有的样子**：读不到就标明，绝不用“做了异常处理”一笔带过。

## 5. 场景五：审查经验卡，防止水文入库

**整套流程的真正瓶颈。多数人会跳过它，知识库三个月后就水了。如果只能守一条纪律，守这条。** ADR 和方案卡都走这一关。

```txt
你现在是“工程经验审查 Agent”。审查下面这条经验是否值得进长期经验库。

硬性门槛（任一不满足直接判不通过）：
A. 删掉所有原项目专有名词后，正文是否仍然完整、自洽、可读？
B. “核心思路 / 决策理由”能否用一句不含形容词的话说清机制或取舍？
C. 把这条经验交给一个没读过原项目的 AI：
   - 若是方案卡，它能否据此写出可运行代码？
   - 若是 ADR，它能否据此在新项目里做出类似架构选择？
D. 来源证据是否只在附录，且标注了是否经过验证？
E. 模糊词扫描：核心结论里是否出现“大概/应该/可能/通常/整体上/做了相应处理”等词？
   出现即判不通过——这是空洞的信号，必须打回重读源码，落到确切值/字段/分支/失败动作。
F. （针对原子规格卡）颗粒度与精确度：
   - 阈值/魔法数字是否都给了确切值？无法确认的是否诚实标了“证据不足”，而非脑补？
   - 分支路径是否每条都有明确去向，没有“做相应处理”式的黑洞？
   - 失败处理是否写明了具体动作（回退/跳过/中断），还是被一笔带过？
   - 最小验证清单是否是可执行断言，而非“测试一下功能正常”？

审查标准：
## 1. 背景是否清楚（这个设计是在什么场景下产生的）
## 2. 问题是否明确
## 3. 约束/适用条件是否说清
## 4. 是否可执行（能据此设计模块、接口、数据结构、流程）
## 5. 是否真正脱离原项目（不看原仓库能否理解并实现）
## 6. ADR 是否说清了“放弃了什么”；方案卡代码是否真实可信、非占位伪代码
## 7. 是否有可溯源的附录证据
## 8. 是否有空泛描述（删除“架构清晰/易扩展/高内聚低耦合/工程化完善”等无机制说明的词）
## 9. 是否可能误导（过度设计、场景不匹配、成本过高）

请输出：
## 审查结论    通过 / 修改后通过 / 不通过
## 主要问题
## 需要补充的证据
## 需要删除的空话
## 需要增强的实现细节
## 修改后的经验卡

要求：严格挑刺，不要客气。宁可少入库，也不要把空泛经验放进长期知识库。
```

## 7. 场景七：维护目录文档

随新增/修改的经验，维护可导航的目录文档。**目录按规模灵活拆分**：经验少时一份总目录就够；多了再按维度拆。文件名沿用中文。

### 拆分阈值（经验之谈，可调）

- **少于约 30 条**：只维护一份总目录 `知识库/经验/总目录.md`，按层级分节列出全部条目即可。
- **超过约 30 条，或某一维度条目密集**：把该维度独立成专门目录文档，总目录退化为"入口页"，只放各专门目录的链接和整体说明。常见维度目录：
  - `知识库/经验/目录-按层级.md`（架构 / 方案 / 原子）
  - `知识库/经验/目录-按领域.md`
  - `知识库/经验/目录-按标签.md`
  - `知识库/经验/目录-按状态.md`
- **原子卡很多时**：在所属方案目录下生成局部目录 `方案目录.md`，列出该方案的各阶段原子卡及其衔接顺序，避免总目录被原子卡淹没。

```txt
你现在是“知识库目录维护 Agent”。根据新增或修改的经验，维护目录文档。

第一步：先判断当前规模，决定目录形态
- 统计现有经验条数。
- 少于约 30 条：只维护一份总目录《知识库/经验/总目录.md》，不要过早拆分。
- 超过约 30 条、或某维度（某领域/某方案的原子卡）条目密集：
  把该维度拆成专门目录文档，总目录改为入口页只留链接。
- 先说明你判断的当前形态，再动手。

第二步：读取本次新增/修改的经验，识别
标题、层级（architecture/pattern/atomic）、parent（所属方案）、领域、标签、状态、related。

第三步：更新目录文档
- 总目录或对应维度目录里，新增/更新该条目的链接与一句话说明（说明必须具体，不许写“相关处理”）。
- 原子卡：除主目录外，更新其所属方案目录下的《方案目录.md》，
  按“阶段1→阶段2→…”顺序列出，并标注阶段之间的衔接约束（如：阶段2 拦截后不进入阶段3）。
- 中文文件名变化时，同步修正所有指向它的链接。

第四步：查重与冲突
- 是否已有高度相似的经验？给出建议：合并 / 保留并标 related / 废弃旧卡（status 改 deprecated）。
- 若发现矛盾结论（两卡对同一问题给了不同做法），先报告，不自行裁决。

第五步：输出变更摘要
列出：新增了哪些条目、改了哪些目录文件、是否触发了拆分、发现的重复或冲突。

要求：
- 不删旧内容，除非明确说明。
- 目录里每条说明必须能让人不点进去就大致知道这条解决什么问题。
- 拆分是为了好查，不是为了好看；不确定要不要拆，就先不拆。
```

## 推荐执行顺序

```txt
0   防偷懒规则
1   项目总纲
2   模块深读
3   流程深读
4A  提炼架构层（ADR）       ┐
4B  提炼方案层（方案卡）     │ 按需选；方案卡是容器
4C  提炼原子层（原子规格卡） ┘ ★默认主力，真正的干货在这里
5   审查              ← 真正的瓶颈，绝不能跳过
6   入库
7   维护目录文档（中文命名，按规模灵活拆分）
12  跨项目归纳共性（读了多个项目后）
—— 以后要用时 ——
8   设计/开发前检索（经验 ≥ 约 20 条后再启用）
9/10 让 AI 编程工具落地
11/13 长期维护与深挖
```

> 实操建议：方案卡只写到“这个方案分几个阶段、阶段之间怎么衔接”，把每个阶段的精确规格全部下沉到 4C 原子规格卡。觉得 AI 还是空——多半是停在了 4B 没往 4C 切。

### 四条纪律

1. **颗粒度砍到原子级**：空洞的根因是范围太大。一张原子卡只回答一个“为什么是这样而不是那样”，精确到字段、参数、阈值、分支、失败动作。还能再切就继续切。
2. **不许用模糊词糊弄**：核心结论里出现“大概/应该/整体上/做了相应处理”即视为没读懂，打回重读。读不到就写“证据不足：需确认 X 是 A 还是 B”，绝不脑补。
3. **经验卡不要一生成就入库**：必须过场景五的审查，否则知识库越来越水。
4. **路径和函数名必须验证**：AI 最危险的失败不是空话，而是“路径正确、函数名虚构”的幻觉——它伪装成有证据，会污染整个知识库。
